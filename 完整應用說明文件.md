# MediPrisma 醫析｜完整應用說明

> 現行產品說明｜基準版本：v0.40.0｜最後核對：2026-07-14
> 本文件以 Markdown 為唯一維護格式；舊比賽交件 PDF 已不再屬於現行文件。

MediPrisma 是一套 client-first 的 SMART on FHIR 臨床資料閱讀與 AI 決策支援工具。它把跨院就診、用藥、檢驗趨勢、報告與臨床文件放在同一工作區，再提供可回查來源的 AI 摘要、安全提醒、Agent 對話、報告解讀、醫療計算機與 IPS 匯出。

> 研究／教學用途，非醫療器材。AI 與計算結果僅供參考；診斷、處置、用藥與追蹤仍應由合格醫療人員依完整病況確認。

## 1. 產品定位

臨床資料常散落在多院所、多次就診與不同 FHIR resource 中。單純把資料「列出來」仍要求使用者自行尋找：最近發生什麼、哪些數值正在變化、有哪些用藥矛盾、下一步需確認什麼。

MediPrisma 的設計目標是：

- **先整合，再生成**：先建立一致的領域資料，再供畫面、工具與 AI 共用。
- **可回查來源**：AI summary 的引用可以跳回原始 FHIR resource。
- **不把缺資料當正常**：資料缺漏、未驗證引用、被截斷或生成失敗都要可見。
- **支援兩種受眾**：醫療人員看臨床密度，民眾看白話說明與行動提醒。
- **把原始資料留在裝置**：完整本地 Bundle 不因載入而上傳；使用 AI 時才傳送當次功能需要的內容。

## 2. 資料來源

### SMART on FHIR

由 EHR 或 SMART Launcher 啟動，使用 public client、OAuth 2.0 authorization code 與 S256 PKCE。App 取得 patient context 後即時讀取 FHIR server，並跟隨 search Bundle 的 next link 讀完整分頁。

Launch：`/smart/launch`
Callback：`/smart/callback`

未知 `iss` 會先顯示確認畫面，避免任意連結把使用者帶往未信任的 FHIR 登入頁。

### 本地 FHIR Bundle

使用者可匯入 `.json` Bundle。解析器支援一般 FHIR R4，並包含：

- 健保存摺／NHI Bridge 常見資料形狀。
- TW-PAS `Claim` 展開。
- Roche DIP／mCODE contained resources 展開。
- `urn:uuid` 與 absolute reference 正規化。
- `MedicationRequest`／`MedicationStatement`、report／observation／encounter 關聯。

Bundle 與內嵌影像以 AES-GCM 加密存於 IndexedDB，session key 只存在目前 tab，最長 12 小時。

### Demo Bundle

Welcome 畫面可一鍵載入內建去識別化示範病人，用於產品展示與測試，不需連接 EHR 或上傳檔案。

## 3. 介面

### Header

- 匯入／清除本地資料。
- Medical／Patient 身份切換。
- 中文／英文切換。
- 登入狀態。
- Overflow menu：主題、版本、連線資訊、回饋等。
- Header 可收合以增加資料空間。

### 左側：臨床資料

目前五個分頁：

1. **病人**：基本資訊、生命徵象、問題清單、預立醫療決定、醫療器材、照護計畫。
2. **就診**：依日期與院所瀏覽 encounter，展開診斷、用藥、檢驗、報告與處置。
3. **報告**：檢驗列表、累積報告、趨勢圖、影像／病理／其他報告與隨選 AI 解讀。
4. **用藥**：用藥歷史，並整合過敏與疫苗子分頁。
5. **文件**：FHIR `Composition` 的人類可讀 narrative，例如 IPS 或出院病摘。

報告支援名稱、院所、內容與結果值搜尋；累積報告把同一 analyte 跨日期 pivot，並對單位與 UCUM 做正規化。

### 右側：臨床功能

目前五個主要功能：

1. **醫療摘要**（預設）
2. **筆記對話**
3. **醫療計算機**
4. **IPS 匯出**
5. **設定**

使用者可 pin／unpin 一般 tab；設定固定保留。桌面與平板使用可調寬雙面板，手機以單欄切換兩側。

## 4. 醫療摘要

Medical Summary 是載入病人後的預設入口。它用固定 Zod schema render 成可重排／隱藏的卡片，不是任意 AI Markdown。

### 標準摘要卡

- 問題清單與推論依據。
- 跨院時間軸。
- 照護提醒與安全警示。
- 待確認／待決策事項。
- 疾病導向檢查趨勢；可跳到累積報告並聚焦 analyte。
- 醫師版用藥核對或民眾版用藥與照護說明。
- Deterministic data coverage。

卡片上方有 sticky jump navigation，會依可見／排序後的卡片同步。每個受眾可保存自己的卡片順序與隱藏狀態。

### 引用設計

App 先從 FHIR 資料建立 source catalog，發給 AI 短 key。模型只能引用 key；日期、機構、resource type 與 resource id 由 app 端解析。點引用會切到左側對應 tab、展開或捲到資源；找不到目標會顯示 fallback，而不是靜默失敗。

「可定位來源」不等於每個自然語言 claim 已經過人工審核。使用者仍需打開原始紀錄確認。

### 自訂摘要

Medical Summary 內有「自訂摘要」視圖。使用者可建立多個 prompt modules、排序、決定是否顯示、選擇 auto-generate，或從 Prompt Gallery 加入。它們與標準摘要分開生成，不會改變固定 clinical schema。

### 生成與快取

Summary 與 Safety 是兩個獨立 slot，由 orchestrator 統一觸發、保留上一份結果、重試失敗部分。結果以 session key 加密快取 12 小時；病人、資料、prompt、model、語言或受眾改變時會使用不同 cache identity。

## 5. Medical Chat Agent

Medical Chat 會依模型能力選擇執行路徑：內建 Agent 模型可以直接回答，也可以依需要呼叫 16 個 FHIR tools 或 1 個 Perplexity 文獻工具。自訂 OpenAI-compatible 模型預設採自動偵測並以標準對話 fail closed；只有以完整正式 FHIR tool schemas 完成不讀取病歷的串流工具往返測試後，才使用深入對話，且只提供瀏覽器綁定的 FHIR tools，不提供外部文獻工具。使用者也可將端點固定為標準對話。切換模型或自訂端點身分時會另開一段空白對話，避免既有內容跨 provider 傳送。

Agent 能力包括：

- 病人資料總覽。
- 查詢／篩選就診與單次就診 detail。
- 診斷、用藥、過敏、疫苗。
- 檢驗、報告、趨勢、異常結果與處置。
- 即時文獻／指引搜尋與 URL citations。
- 最多 10 個 AI SDK steps，必要時進入 follow-up／synthesis round。
- Streaming idle timeout、停止按鈕與病人切換 abort。

其他對話工具：

- `/` slash templates 與管理 drawer。
- Prompt Gallery。
- 可編輯 system prompt。
- 圖片上傳與媒體 consent。
- Whisper 語音轉錄。
- Reply-to、複製、next-step suggestions。
- 全螢幕閱讀。

登入且非無痕時，文字對話 debounce 後存 Firestore；圖片不保存。訪客與無痕對話只在目前記憶體。

## 6. 報告 AI 解讀

影像、病理、出院病摘或其他文字報告可隨選產生：

- 忠實翻譯／整理。
- 重要發現。
- 白話解釋與注意事項。

原文保留供對照，解讀結果使用 12 小時加密 cache。此功能只在使用者按下生成時消耗額度。

## 7. 醫療計算機

目前有 57 個 calculator definitions，分為 10 類：一般、心臟、腎臟、肝臟、肺部、神經、血液、電解質、腸胃、心理／評分等。

特點：

- MDCalc 風格的搜尋、收藏與 detail。
- 顯示適用時機、輸入單位、公式與注意事項。
- 依 canonical analyte、LOINC、specimen 與 unit dimension 從目前病人報告自動帶值。
- 找不到或不確定時保留手動輸入，不用錯誤檢體猜值。
- 結果可複製。

計算機只做公式運算與資料帶入，不應直接當作診斷或治療建議。

## 8. IPS 匯出

IPS feature 建立 FHIR document Bundle，主要包含：

- Composition
- Patient
- AllergyIntolerance
- MedicationStatement
- Condition
- Immunization
- Procedure
- Observation／DiagnosticReport
- Device／DeviceUseStatement
- CarePlan
- Consent

使用者可調整匯出 scope、預覽 Markdown／JSON 並執行 IPS-lite validation。AI 可從摘要 evidence 推論缺少的 problem candidates，但每一項都必須經使用者確認才納入，並保留來源／推論標記。

## 9. Prompt Gallery 與個人範本

共享 prompt 可標示 `chat`、`summary` 或兩者，並依 category、specialty、audience、tags 分類。使用者可搜尋、排序、預覽、分享與管理自己的 prompt。

個人 chat templates 與 custom summary modules 在登入時同步 Firestore，訪客則留在 localStorage。Shared prompt 與個人副本分開，使用後修改個人副本不會改寫社群原件。

## 10. 雙受眾與多語言

首次進入可選 Medical 或 Patient：

- Medical：保留 resource type、臨床術語與高密度結構。
- Patient：使用白話標籤、較友善的用藥／提醒語氣與不同 summary schema 指示。

身份可以隨時切換；summary、templates 與 card layout 會依 audience 分開。App 支援繁體中文與英文、深色模式與字級設定。

## 11. AI 模型與額度

模型清單以 `src/shared/constants/ai-models.constants.ts` 為準。

| Provider | 免費 proxy-eligible | 需自備 key |
|---|---|---|
| OpenAI | GPT-5.4 Nano | GPT-5.6 Luna、GPT-5.6 Terra、GPT-5.6 Sol |
| Google | Gemini 3.1 Flash-Lite、Gemini 3 Flash Preview（預設） | Gemini 3.5 Flash、Gemini 3.1 Pro Preview |
| Anthropic | Claude Haiku 4.5 | Claude Sonnet 4.6、Claude Opus 4.8 |

免費模式需要可用的 Firebase 匿名或登入 session，並由後端執行每日 quota。自備 key 只在瀏覽器 direct-provider path 使用。若 premium model 沒有對應 key，程式會降級到免費 default 並顯示實際執行模型。

Chat、Medical Summary 與自訂摘要各自保存 model preference，避免一個功能的選擇意外改變其他背景生成。

## 12. 隱私與安全

主要控制：

- SMART public client + S256 PKCE，無 client secret。
- Bundle／影像 AES-GCM、session key、12 小時上限。
- AI cache 加密且不可跨 tab 解密。
- API keys 預設 sessionStorage；明確選擇才長期保存。
- Proxy 使用 Firebase ID token 與可選 App Check。
- Tool payload 與 user-message copy 做 best-effort PII minimization。
- DOMPurify、structured schema、feedback escape／origin／rate limit。
- CI、ESLint architecture boundaries、CodeQL、Dependabot。

剩餘風險：client-side encryption 不能防同 origin XSS／惡意 extension；自由文字 scrub 不能保證匿名；Firestore chat 可能含 PHI；static host 不會套用 Next response headers。詳見 [SECURITY.md](./docs/SECURITY.md) 與 [PRIVACY_POLICY.md](./PRIVACY_POLICY.md)。

## 14. 開發與部署

需要 Node.js 20+；CI 使用 Node 24。

```bash
npm install
npm run dev             # Turbopack, http://localhost:3001
npm run dev:webpack     # webpack, http://localhost:3000

npx tsc --noEmit
npm run lint
npm test
npm run test:e2e

npm run build
npm run build:gh
npm run build:mediprisma
```

GitHub Pages 是 static export：沒有 Next `/api/feedback`，也不會執行 `next.config.ts.headers()`。正式回饋需外部 endpoint；CSP／HSTS 等 header 需由實際 CDN／host 提供。

Master 上 CI 通過後自動部署 GitHub Pages，並可同步相同 app 到 mediprisma.tw `/app`。Firebase Functions／Rules 在獨立後端 repo。

## 15. 測試與品質

- 近 200 個 Jest／Playwright test files；確切清單以 `__tests__/` 與 `e2e/` 為準。
- Jest coverage ratchet：branch 18%、function 21%、line／statement 23%。
- Playwright client suite 使用 synthetic Bundle 與攔截的 AI stream。
- Emulator suite 驗匿名登入、ID token 與 proxy auth chain。
- CI gate：typecheck、lint、Jest、GitHub Pages build。
- E2E workflow 與 CodeQL 分開執行。

本機 loop verifier 可執行 typecheck、lint、test 與選用 build，並把 structured verdict 寫到 `.loop/`。

## 16. 使用流程

### 快速了解病人

1. SMART launch、匯入 Bundle 或載入 Demo。
2. 確認 Medical／Patient 身份。
3. 先閱讀右側 Medical Summary。
4. 點引用回查左側原始紀錄。
5. 需要更多細節時看累積報告／單次就診。

### 回答臨床問題

1. 在 Medical Chat 輸入具體問題。
2. 觀察 Agent 呼叫了哪些 tools。
3. 文獻問題確認有真實 URL citations。
4. 回答中的病人事實回到 FHIR 原始紀錄驗證。
5. 複製前由醫療人員校對。

### 產生可攜摘要

1. 開啟 IPS Export。
2. 選擇資料 scope。
3. 檢查 AI inferred problems，逐項接受／拒絕。
4. 閱讀 Markdown preview 與 validator 結果。
5. 下載／複製 FHIR Bundle，交由目標系統再驗證。
